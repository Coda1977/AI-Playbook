# One-Page Plan: Synthesis Feature

**Date:** 2026-05-06
**Status:** Spec, awaiting review
**Author:** Yonatan (with Claude)

## Why this exists

Workshop participants reach the Review page and feel overwhelmed. Twenty-plus starred items, distributed across six categories of use cases and five rules of change strategy, do not read as a plan. They read as a checklist. A workshop participant explicitly asked: "add another step that creates one narrative from your plan and clusters your best ideas as one plan."

The synthesis feature gives the manager a single, editorial, one-page plan that walks them out of the workshop with a story rather than a list.

## What it is

A new phase, `synthesis`, that follows the existing `commitment` (Review) phase. The user clicks a primary CTA on the Review page labeled **"Generate My Plan (Experimental)"**. The AI clusters the manager's intake, use cases, and change actions into one or two storylines, and produces a structured document. The user lands on a dedicated page that renders the plan with the same visual language as the rest of the app, with options to export to Word or PDF.

The synthesis is generated once. It cannot be regenerated from inside the app. The only way to produce a new synthesis is to start over from intake.

## Output shape

The plan has four parts:

1. **Title.** A single-sentence headline that names the manager's central tension or wedge. Generated by the AI.
2. **Lede.** A short paragraph (roughly 60 to 100 words) that frames the plan's central thesis. Synthesizes from intake (failure risks, success vision, fluency gap) and the manager's starred items together. When intake is thin, the starred items carry more weight. When intake is rich, intake leads.
3. **Storylines.** One or two storylines, never more. The AI decides whether the manager's plan supports two distinct themes or a single tighter one. Each storyline contains:
   - An eyebrow label, format `Storyline 0X · {short theme name}`
   - A headline, the storyline's central claim, written as a sentence
   - A thesis block, one or two sentences that explicitly state what bet this storyline represents
   - Prose, two short paragraphs that synthesize the manager's items into a coherent argument
   - An "AI Use Cases" list, the use cases that genuinely support this storyline
   - A "Change Actions" list, the change actions that genuinely support this storyline
4. **This Week.** Three concrete starting actions, distilled from the plan, written as imperative sentences. Each starts with a verb in bold.

### Item selection rules

- The AI receives all intake fields, all use cases (starred and unstarred), and all change actions (starred and unstarred).
- Starred items are weighted more heavily as signals of priority, but unstarred items can appear in the synthesis if they meaningfully support a storyline thesis.
- Each storyline pulls only the items that genuinely belong. There is no requirement to surface every starred item. Items not surfaced in the synthesis remain visible on the existing Review page, which serves as the audit trail.
- Item lists per storyline are not capped at a fixed count. The AI judges fit.

## UX flow

### Trigger

The Review page (CommitmentView) gets a new primary CTA placed at the top of the page, immediately after the hero block and before the existing stats grid. The CTA is rendered as a styled card with brief explainer copy and a single button. A secondary echo of the same button appears at the bottom of the Review page in the existing action button row, for users who scroll past the top.

When `state.synthesis` is empty:
- Button label: **"Generate My Plan (Experimental)"**
- Click triggers `generateSynthesis()`, which sets `phase: "generating-synthesis"` and calls the API
- A `GeneratingIndicator` view (mode `synthesis`) shows during generation, mirroring the existing primitives and playbook generation views

When `state.synthesis` is populated:
- Button label: **"View My Plan"**
- The "(Experimental)" tag is dropped, since the user has already seen the artifact
- Click navigates directly to `phase: "synthesis"` without an API call

### Synthesis page

A new full-page view, `SynthesisView`. The page renders `state.synthesis` using the visual language documented below. The page contains:

- A hero block (title, eyebrow, meta line)
- The lede card
- One or two storyline cards
- The "This Week" card
- An action row with three buttons: "Back to Review" (ghost), "Export Word" (ghost), "Download PDF" (primary)

There is no edit-in-place. Users who want to tweak language export to Word and edit there.

### Failure handling

If the API call fails, the user stays on the Review page. An `ErrorBanner` shows the same shape as the existing playbook generation failure: "Something went wrong while writing your plan. This usually means a connection issue." The CTA remains "Generate My Plan (Experimental)" so they can retry. Failed attempts do not lock the user out. Only one *successful* generation locks the synthesis.

### Regeneration

There is no in-app regenerate button. The synthesis is a one-shot artifact. To produce a new synthesis, the user must use the existing "Start Over" flow which clears all state and returns to intake. This deliberately raises the cost of regeneration to discourage users from rolling dice until they like a phrase.

If the user changes starred items after the synthesis is generated, no warning appears. The existing Review page remains as the live audit surface; the synthesis is a frozen artifact tied to the moment it was generated.

### Reducer cases

Three new actions:

- `SET_SYNTHESIS` with `synthesis` payload, sets `state.synthesis` and `state.phase = "synthesis"`
- `SET_PHASE` already exists, used to navigate to and from the synthesis page
- `RESET` already clears state via the existing flow, no change needed

The `synthesis` field persists to localStorage like everything else, via the existing `saveState` effect.

## API endpoint

New file: `api/synthesis-generate.js`. Same shape as `playbook-generate.js`.

- HTTP method: POST
- Body: `{ intake, primitives, plan }`. The full intake plus the full primitives and plan trees, including starred flags.
- Model: `claude-sonnet-4-6` (matches existing endpoints)
- `max_tokens`: 3000 (synthesis output is roughly 1000 tokens; 3000 leaves headroom for variance)
- Response: `{ synthesis: { ... } }` with the JSON shape documented above

### Prompt structure

The system prompt must establish:

1. **Role.** The model is an editorial synthesizer, not a list-builder. It produces an opinionated one-page plan, not a redistribution of inputs.
2. **The two-source rule.** Synthesize the thesis from whichever signal is stronger, intake or starred items. If intake is rich, weight it heavily. If intake is thin, let the starred patterns lead.
3. **Stars are signal, not filter.** Items the manager starred carry more weight as priority signals, but unstarred items can appear in a storyline if they genuinely support it. Items that do not fit a storyline are simply omitted.
4. **One or two storylines, no more.** The AI must judge whether a single thesis fits the data or whether two distinct storylines emerge. Three is forbidden.
5. **Each storyline must have a real thesis.** The thesis sentence must be falsifiable, pointed, and distinct from the other storyline's thesis. Generic statements like "AI helps your team" are not allowed.
6. **No invented experiences.** Same anti-hallucination rule used in the existing `playbook-generate.js` and `chat.js`. The manager's specific situation drives the plan; the AI must never fabricate metrics, outcomes, or stories.
7. **No em dashes in any output copy.** Use periods, commas, colons, semicolons, or parentheses. This is a project-wide rule.
8. **Output is JSON only**, in the schema documented above.

### Quality checks

The prompt should include explicit quality checks the model self-applies before returning:

- Could this plan belong to anyone, or is it specific to this manager? If anyone's, rewrite it.
- Is each storyline's thesis distinct from the lede and from the other storyline's thesis?
- Does each storyline include both prose and source items, with the prose synthesizing rather than restating?
- Are all "This Week" actions concrete enough that the manager knows what to do Monday morning?

## Export

Add to `src/utils/export.js`:

- `exportSynthesisDocx(state)`. Generates a Word document from `state.synthesis`. Mirrors the structure of `exportPrimitivesDocx` and `exportPlaybookDocx`, using the `docx` library. Document layout: title heading, meta line, lede paragraph, then for each storyline a heading, thesis (italic), prose paragraphs, "AI Use Cases" subheading + bulleted list, "Change Actions" subheading + bulleted list. Then a "This Week" heading and numbered list.
- PDF export uses `window.print()`, same as existing flows. New print rules in `index.css` ensure the synthesis page prints cleanly: hide nav and action buttons, ensure black hero prints with `print-color-adjust: exact`, force white background.

## Out of scope

The following are explicitly not part of this feature:

- In-place editing of the synthesis (users export to Word and edit there)
- In-app regeneration (users start over from intake)
- Multiple synthesis variants (one storyline count is decided by the AI, not the user)
- Sharing the synthesis via URL or email
- A/B-testing different prompt structures
- Per-section regeneration (regenerate only Storyline 02, etc.)
- Manual storyline editing (the AI's clustering is the artifact)

These are all candidates for future iterations once the experimental version is in users' hands.

## Success criteria

For the experimental release to be considered successful:

- A workshop participant can generate, view, and export a synthesis without errors
- The synthesis output is judged by the workshop facilitator (Yonatan) as feeling like one cohesive plan rather than a redistribution of items
- The synthesis page renders cleanly on desktop and degrades acceptably to mobile
- Print export (PDF) produces a readable document
- Word export opens correctly in Microsoft Word and Google Docs
- API failures are handled gracefully without lost state

## Risks

1. **Thesis quality.** The entire feature hinges on the AI writing a real, falsifiable thesis sentence per storyline. If the model writes flabby theses ("AI helps your team work better"), the storylines collapse into mixed buckets. The prompt must demand specificity and the model must be willing to challenge the manager's framing.
2. **Variance.** Different runs of the same data may produce different storyline counts and different items per storyline. The "no regenerate" rule means users live with whatever lands. This is intentional but bears watching.
3. **API cost and latency.** With `max_tokens: 3000` and the full intake-plus-trees as input, this is the most expensive call in the app. Acceptable for an experimental feature, worth monitoring at scale.
4. **Mobile rendering.** The two-column evidence grid is the most fragile layout element. Tested via the 768px breakpoint rule but real device testing required.

## Decisions log

- **Cap at 2 storylines, allow 1.** Workshop feedback is "overwhelming." Two storylines forces synthesis; three lets the AI cheat.
- **Stars are signal, not filter.** When intake is thin, items must carry the thesis. A starred-only filter would fail thin-intake users.
- **One generation, no regenerate.** Eliminates dice-rolling. The synthesis is editorial, not a vending machine.
- **Synthesis lives as a new phase, not a modal.** It is a real artifact that needs its own page for printing, exporting, and reading.
- **Gold restricted to star icons only on this page.** Gold is the "favorite" color in the app's vocabulary. Using it as a section accent dilutes its meaning.
- **No em dashes anywhere.** CLAUDE.md project rule. Strictly enforced in prompt and rendered output.
