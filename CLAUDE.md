# AI Playbook - Project Guidelines

## Overview

Workshop tool: managers answer 7 intake questions, AI discovers personalized use cases (Phase 2), then builds a change strategy grounded in behavioral science (Phase 3), then review & export (Phase 4).

## Tech Stack

React 19 + Vite 7 + Tailwind CSS 4 + Vercel Serverless Functions. Claude Sonnet 4.6 API for generation. `docx` + `file-saver` for Word export. localStorage for persistence.

## Design System

- **Aesthetic**: Editorial-calm, Miro-derived DNA -- white canvas, ink text, one salmon accent, one star-blue accent. Reduced palette; no black/white/red.
- **Typography**: Hanken Grotesk only (weights 400-700; self-hosted variable font, Google Fonts as fallback source). No Montserrat, no Roboto Condensed.
- **Palette**: `#FFFFFF` canvas, `#1C1C1E` ink (text, primary pill CTAs, dark surfaces), `#FFB5AB` accent/salmon ("the product speaking": brand mark, kickers/eyebrows, generating progress, Big Move numerals, done-step checks), `#ACD8FF` star ("the user's commitments," appears nowhere else: star buttons, starred-card borders, tray), `#C6453C` danger (destructive/error states ONLY, never decorative). Full token list lives in `src/index.css` (`@theme`) and the `C` object in `src/config/constants.js` -- see Pitfalls.
- **Bones**: rail-focus-tray boards for Phase 2/3 (`BoardRail` + focus panel + `CommitmentTray`) -- one category/rule in focus at a time, starred items accumulate in an always-visible tray; a single shared `GateBar` is the ONLY home for primary actions on working screens ("the tray displays, the gate bar acts"); light checklist `GeneratingIndicator` screens (no dark interstitials mid-flow); a dark full-page poster ONLY on the Big Move (`SynthesisView`), with a light print variant for PDF/print export.
- Paper grain disabled. Card-based layout, hairline borders over shadows (shadows reserved for overlays: modal, mobile tray bottom-sheet).

## Critical Constraints

**Phase names**: "AI Use Cases" (not Discovery/Primitives) and "Change Strategy" (not Playbook/Change Management).

**Chat brevity** (`api/chat.js`): 60-word target stated once per system prompt; max_tokens 500 is the hard ceiling. (Chat returns structured `tool_use` output since June 2026, so the old dual-format token tuning -- 250/400 -- no longer applies.) Verbosity creep is the #1 issue -- preserve constraints when modifying prompts. Adding more prompt text makes verbosity WORSE (model mirrors input energy). Use max_tokens as the hard ceiling, not prompt instructions.

**Behavioral science**: The 5 rules (Start at End, Make It Safe, Script Steps, Start Small, Make Progress Visible) are research-grounded. Don't dilute the prompts in `playbook-generate.js`.

**Validation**: Word-count thresholds (0/5/15/16+), not character count.

**Regeneration**: ConfirmModal in App.jsx warns before replacing existing primitives.

## Pitfalls

- Never use `transition: all` on `.canvas-rules` (breaks layout on chat panel close)
- Never use any font besides Hanken Grotesk (no Montserrat, no Roboto Condensed)
- Never change phase names back to "Discovery" or "Playbook"
- Never break the 4-phase linear flow: Intake -> Use Cases -> Strategy -> Review
- Keep state simple: Context + localStorage, no Redux
- Keep animations subtle (0.3-0.5s, transform/opacity only); `prefers-reduced-motion` must disable bloom/entrance/spinner/pulse animations
- `GateBar` must stay a direct child of `.canvas-rules` (its sticky-to-viewport-bottom behavior depends on that position in the DOM)
- Colors live in BOTH `src/index.css` (the `@theme` block) AND the `C` object in `src/config/constants.js` -- change them together or the two drift

## AI Behavior (merged from `improving-ai` branch)

### How the AI works

Four API endpoints in `api/`:
- **`primitives-generate.js`** -- Initial use case generation (Phase 2). One-shot, no chat. Generates 12 ideas total, selected by EXPECTED VALUE for the manager (recurrence x reach, pain, vision proximity, adoption probability; surprise breaks ties), with FOUR committed BIG SWINGS per list (required `bigSwings` schema field, logged server-side): transformative reshapes INSIDE realms the manager named, each stating its first probe. The fluency floor constrains an idea's first probe, never its destination's size. Structural enforcement (July 2026): the model commits to `focusCategories` (3 ideas each) and `stretchCategory` (1) and the server trims to the committed counts; the prompt receives only THIS team's fluency floor via `teamFluencyFloor()` in lib/workshop.js; forced `modalityCoverage` fields guarantee assistant/critique/role-play/transcription spread (assistant family conditional on someone being able to set one up); failureRisks is in the prompt. Returns structured output via forced `tool_use` (`submit_use_cases`), migrated June 2026. max_tokens 2048.
- **`playbook-generate.js`** -- Initial strategy generation (Phase 3). One-shot, no chat. 2 actions per rule, 3 on the 1-2 rules the intake signals prioritize; the model commits via a `prioritizedRules` schema field and the server ENFORCES the counts by trimming (the model overshoots prose count limits, so the trim is the contract; July 2026). DECOUPLED from Phase 2 (July 2026): starred use cases are optional context only (name one in a few words when it sharpens an action; never restate, never deliver, never build the plan around them); the plan is intake-fueled change leadership (Rule 1 speaks in the vision's own finish lines, Rule 2 names the intake's actual fear, Rules 3-5 scope from real rituals), with an explicit anti-template rule (two managers with different intakes must get visibly different plans). Move palette as value preference (defaults beat broadcasts for Rules 3-5, meetings reserved for fear-naming); every action's subject is a person with the AI workflow only as object (the default changes, the artifact does not get built); the change audience is the manager's OWN team, never the clients/users the team serves. THREE committed BOLD MOVES per plan (required `boldMoves` field, logged server-side): each changes an operating default the team runs on today (a routine, process, deliverable form, or ownership) with bounded but genuine risk; accepted risk is what commitment looks like; bold never means broad and never tramples the intake's named fear. max_tokens 4096. Returns structured output via Anthropic `tool_use` (forced `tool_choice`). Has quality check #5 (anti-hallucination).
- **`synthesis-generate.js`** -- One-page plan synthesis (Phase 4). One-shot, no chat. Single-thread discipline via a required `leftOut` commitment field (every starred item lands in actions if it serves the title's thread, or in leftOut with a reason; a thread is ONE workflow family and a title needing "and" is two threads; leftOut is logged server-side and stripped from the response). Every action is a human move: never starts with Build/Create/Set up, no two actions cover the same work, actions belong to the manager's own team. Returns structured output via Anthropic `tool_use` (forced `tool_choice` on `submit_one_page_plan`). max_tokens 4096.
- **`chat.js`** -- Ongoing chat for both phases. Two system prompt builders: `buildPrimitivesSystem()` for use cases, `buildPlaybookSystem()` for strategy. BIG MODE (July 2026): the server detects big/bolder/huge in the user message (`BIG_ASK` regex) and injects a one-turn destination-framed block (what the work IS changes; variety defined relative to the live board, never a shape menu, which caused an inversion rut; fewer-but-bigger allowed). max_tokens 500. Returns structured output via forced `tool_use` (`reply_with_ideas` tool with `content` + `ideas` fields), migrated June 2026 from the dual prose + `---IDEAS---` format whose parsing could silently drop the idea cards.

All four endpoints now use forced `tool_use` for structured output. User-provided text (intake fields, idea/action lists) is wrapped in XML-style tags (`<manager_profile>`, `<current_ideas>`, etc.) in every prompt to keep data distinct from instructions. All four reject cross-origin POSTs via `lib/apiGuard.js` (ops scripts must send an `Origin` header matching the deployment host).

### Changes made (July 2026, feat/quality-pass)

- **Bold moves (2026-07-06, Yonatan's definition after two critique rounds).** Both knowledge experiments failed first: his full 5-rules article as the prompt's knowledge base lost to the condensed hints in BOTH configurations (freeform 0-3-1, article-in-engineering a noise-level tie over two runs), proving the condensed promptHints carry the methodology efficiently and that neither knowledge nor freedom was the freshness lever. What worked: three committed `boldMoves` per plan, defined by him as changing an operating default the team runs on today with bounded but genuine risk (rejected framings: publicity as skin-in-the-game, retreat-path removal, power handoffs, sacred rituals). Acid tests in the prompt: the dinner question, the slight-nervousness question. Chat's playbook big mode rewritten to the same definition. Verified vs production with a boldness judge dimension: 3 wins / 5 ties / 0 losses overall across two runs, boldness dimension won 7 of 8. First measured improvement over the shipped Phase 3.
- **Phase 3 decoupled from Phase 2 (2026-07-05/06, Yonatan's two-apps framing).** Use cases jog imagination; the change plan is the behavioral science of leading a team to a new future. The 7-9/12 pilot dominance turned out to be the model obeying our own anchor-roles instruction, so the fix was subtraction: pilot selection, anchor placement, and star routing deleted; starred items demoted to optional naming material; creative latitude fueled by the intake added, plus a cross-persona template check in the journey harness (near-identical actions across personas; baseline: 1 pair in 36 actions). Playbook suite hit 5/5 twice consecutively for the first time. The dominance COUNT stays 7-9/12 but its source moved from starred-anchoring to the manager's own vision, and the actions are leadership moves, not workflow delivery; if less vision-flavored concentration is ever wanted, the lever is the intake's vision question, not this prompt.
- **Altitude and audience refinement round (2026-07-04/05, from Yonatan's live walkthroughs).** His OD run showed intake grounding creating tunnel vision (one starred idea colonized 7/13 plan actions and the Big Move) and that the complaint was BIGNESS within the manager's named realms, not realm coverage. Shipped: four committed `bigSwings` per idea list (schema field, not prose quota); the fluency floor explicitly constrains an idea's first probe, never its destination's size ("Capable" had been silently shrinking ambition); Phase 2/3 disconnect (starred use cases are the CONTENT of the change, the five rules are the engine, every action's subject is a person); audience lock (change audience is the manager's own team, never clients/users, discovered on a coach persona whose "team" was 2 fellow coaches and whose 30 coached leaders were clients); "go bigger" chat mode with server-side keyword detection. Big mode originally shipped with a four-shape menu, which anchored the model on inversion in four consecutive chats; rewritten destination-framed with variety defined relative to the live board. Big Move single-thread discipline via a required `leftOut` field after prose "omit what doesn't fit" failed three times; the model's first counter-move was widening the title to cover both threads, closed by defining a thread as ONE workflow family. Verified across three full persona walkthroughs (OD director, internal coach, PMM) plus a programmatic sales-director journey.
- **Diverge/converge pipeline: built, measured, dropped (2026-07-03).** A two-call Phase 2 (unconstrained 25-30 candidate longlist, then value-based selection) was fully built and A/B-tested against the single-pass prompt (Opus judge, order-swapped pairwise, 4 runs, 4 personas incl. an OD director): absolute scores tied 4 vs 4, pairwise 1 win / 9 losses / 3 ties for the pipeline, all slight margins, and converge kept selecting accelerator-shaped lists (no transformative ideas surfaced even at Capable/Capable) while occasionally passing floor-brushing candidates from the unconstrained pool. Dropped for its double latency and complexity; Yonatan's call on the evidence. Everything structural the pipeline work proved out was folded into the single-pass prompt and verified at parity with the frozen baseline (3 ties, 1 variance loss; suite green).
- **Phase 2 structural enforcement (kept from the pipeline work).** Transformative license replaces the old smallness instructions (root cause of "small ideas": the prompt instructed smallness four ways); expected-value selection criteria; `teamFluencyFloor()` injects only the applicable fluency tier (asking the model to map a four-tier catalog is where the floor kept leaking); downscope-to-first-probe rule (an over-fluency idea becomes the version the team can run today); failure-risk floor names the safe input form, and perception-shaped risks ("smells like AI reading employee data") exclude the data entirely; `focusCategories`/`stretchCategory` commitment + server trim (counts overshot to 15 under prose); forced `modalityCoverage` killed the paste-monoculture (percentage instructions did not); below-Adoptive ideas may never use self-running verbs (auto-generate, trigger, sync).
- **Playbook move-palette as a value preference (no shape quotas).** Five move types (conversation, structural default, practice field, social proof, measurement rhythm); pick the lever with the highest behavior-change probability per unit of manager effort; defaults beat announcements for Rules 3-5; meetings reserved for fear-naming; levers must be operationalized, never generic hygiene; stock moves fit-checked (a not-yet-started team has no bright spot to find yet); Rule 2 protects the loss the intake actually names (judgment atrophy, brand voice, privacy), not the stock fear-of-failure set.
- **Transformative star routing (playbook).** A starred use case that reshapes how work happens becomes Rule 1 destination material and its first probe IS the Rule 4 pilot, outranking accelerator stars for the pilot slot (precedence had to be explicit or the accelerator won the slot).
- **Chat prompt updates.** Primitives chat: suggestions may be accelerator or transformative (transformative names its first probe); pushback must change task/modality/altitude, never vary the rejected idea. Playbook chat: one move-palette line. The chat-primitives eval snapshot had been stale (missing the live modality line); synced.
- **Eval infra learnings.** Rubrics judge VALUE, never shapes: dominance counting was removed from rubrics (prompt-resistant, measured in quality-ab instead), the methodology's own moves (leader demo, loss-naming 1:1s, scripted Monday instructions, start-with-the-willing) must never be flaggable as hygiene/broadcasts, and a human paste-habit on a rhythm is not a pipeline. The promptfoo config mirrors the server trim via a transform so assertions test what clients receive. Word-cap asserts fail only on 2+ offenders or 35+ words (single stragglers are stochastic). Known flake: the promptfoo grader occasionally fails JSON extraction on llm-rubric; rerun before diagnosing. Playbook suite passes ~3-4/4 per run with rotating single-generation variance; treat the value rubric as an instrument, not a hard per-run gate.
- **Quotas redistributed within the lighter-output budget** -- primitives: 12 ideas total, 1-3 per category weighted by fit (the exactly-2 quota forced filler in stretch categories); playbook: 2 per rule + 3 on the 1-2 prioritized rules, server-enforced via trim. The prompt's prioritization instructions were previously unexpressible under the fixed schema.

Previously suggested ideas are serialized back into replayed assistant turns as `[Suggested with this reply: ...]` so the model stops re-suggesting them; both prompts instruct against re-suggesting. Starred use cases reach playbook chat as category titles, not raw ids.

### Changes made (June 2026)

- **`chat.js` migrated to `tool_use`** -- The last text-mode parser. Forced `reply_with_ideas` tool returns `{content, ideas}` as a guaranteed structure; the `---IDEAS---` separator parsing and all three fallback paths were deleted. Parse failures used to silently drop idea cards (coaching text arrived, addable suggestions didn't). max_tokens 400 -> 500 for tool-call overhead. Verified live in both modes.

The old line was never deleted, so the model resolved the conflict unpredictably per turn. Kept "different angle."
- **Synthesis examples no longer teach invented deadlines** -- Both good `bigMoveTitle` examples contained timeframes ("within six weeks," "by July") the manager never stated, training the model to fabricate deadlines on the most visible line of the plan. - **User text wrapped in XML-style tags in all four prompts** -- Intake fields and idea/action lists now sit inside `<manager_profile>`, `<current_ideas>`, `<current_actions>`, `<all_actions>`, `<starred_use_cases>`, `<ai_use_cases>`, `<change_actions>` so free-text answers can't bleed into instructions; mild prompt-injection resistance for free.

### Changes made (May 2026)

- **`playbook-generate.js` migrated to `tool_use`** -- Previously asked the model for a JSON object in free text and ran `JSON.parse` on a regex-extracted substring. Failed intermittently under concurrent load (workshop scenarios: ~3/10 users bounced from Phase 3 back to Phase 2 with "connection issue" toast). Root cause was the contract, not the parser: any preamble, trailing prose, or stray brace from the model broke parsing. Replaced with a forced `tool_use` call against a JSON Schema for the plan shape. The API guarantees a structured `input` object on the `tool_use` content block, so the entire parsing layer (regex + `JSON.parse` + brace handling) was deleted. Same prompt, same model, same idea quality; reliable transport. Verified 10/10 concurrent API calls and 5/5 real-browser end-to-end runs after deploy.
- **`playbook-generate.js` max_tokens 2048 -> 4096** -- Headroom for any preamble Claude generates alongside the tool call. You only pay for tokens generated, not the ceiling.
- **`playbook-generate.js` error logging** -- On the rare path where `tool_use` is missing from the response, log `stop_reason`, the list of returned content block types, and the first 1000 chars of the raw response. Future failures surface ground truth in Vercel logs instead of forcing another investigation cycle.
- **`synthesis-generate.js` migrated to `tool_use`** -- Mirror of the `playbook-generate.js` migration. The synthesis endpoint was the only remaining HARD ERROR path (free-text `JSON.parse` with no fallback → HTTP 500 → user bounced back to Review with a generic toast). Now uses a `submit_one_page_plan` tool with a JSON Schema covering title/lede/storylines (1-2)/thisWeek (exactly 3). Same prompt, same model. max_tokens bumped 3000 -> 4096 for tool-call preamble headroom. (Schema since replaced by `bigMoveTitle` + `actions` (4-6) in the June 2026 lighter-output release.)
- **localStorage quota handling** -- `saveState()` in `src/utils/storage.js` previously silently swallowed `QuotaExceededError`, so users with long chats could lose work on refresh without any signal. Now detects quota errors (cross-browser: `QuotaExceededError`, `NS_ERROR_DOM_QUOTA_REACHED`, code 22/1014) and dispatches a `STORAGE_QUOTA_EVENT` window event. `App.jsx` listens for it and surfaces a toast pointing the user at the Word export. Reducer also caps each chat thread at 30 messages, preserving the opener (index 0) and trimming the tail. This is the only behavior change visible to long-running power users.
- **Retry UX on generation failures** -- `ErrorBanner` always had an `onRetry` prop but it was never wired. `App.jsx` now keeps a `lastFailureRef` of the last failed generation (kind + args) and exposes a `handleRetry`. Banner shows on Phase 3 as well (was previously skipped). Error copy distinguishes timeout / network / 5xx server / 4xx client via the new `describeApiError` helper in `src/utils/api.js`.

### Changes made (Feb 2025)

- **max_tokens tuned to 250** -- Dropped from 512. Tested at 200 (too tight, cuts off JSON ideas), 250 is the sweet spot. This is the only reliable brevity control -- prompt instructions alone don't work.
- **Static chat openers** -- `ChatDrawer.jsx` no longer fires an API call on open. Shows instant static greeting ("Let's explore [topic]. What would be most useful to dig into?"). User speaks first, zero latency.
- **Chat angle diversity** -- Closing questions in both `buildPrimitivesSystem` and `buildPlaybookSystem` instruct the AI to pivot to a different angle, not drill deeper into the same direction.
- **Anti-hallucination constraint** -- Added to `chat.js` (both prompt builders, instruction #5) and `playbook-generate.js` (quality check #5). AI must never fabricate experiences, metrics, or outcomes. If suggesting they share a story, script the format ("share your experience"), not the content ("explain how automating X saved Y hours").
- **Mission context** -- Both chat system prompts open with a one-line workshop context so the AI understands the bigger journey (intake -> use cases -> strategy).

### Key learnings for future AI work

- **max_tokens is the only hard brevity control.** Prompt instructions ("60 words max! Count them!") are unreliable. The model estimates, doesn't count.
- **Long system prompts cause verbose responses.** The model mirrors input energy. To get shorter answers, trim the prompt rather than adding "be brief" instructions on top.
- **The AI fabricates personal anecdotes** when the prompt asks for specificity but the intake data doesn't include actual experiences. Fix by constraining output ("never invent"), not by adding more intake fields.
- **200 max_tokens is too tight** for the dual-format response (prose + JSON). The model writes good 60-word prose but runs out of tokens before the ideas JSON. 250 works. 512 gives too much runway.
- **Static openers > auto-generated first messages.** Removing the initial API call on chat open eliminates 4-6 seconds of latency and lets the user drive the conversation direction.
- **A diverge/converge second pass is not the quality lever either.** Two-call generation (wide longlist, then value-based selection) measured at parity with a good single-pass prompt and never surfaced the transformative ideas it existed for, while adding latency and floor-brushing risk (2026-07-03, Opus judge). The levers that DO move quality: what the prompt licenses (removing smallness instructions), deterministic prompt assembly, and grounding. The strongest untried lever remains the curated gold use-case library.
- **Inject only the applicable rule, don't ship the catalog.** The fluency floor held only after the server started injecting the single tier matching this team (`teamFluencyFloor()`); with all four tiers in the prompt the model kept misapplying them. Same lesson as chat.js shipping only the current rule's science.
- **Menus anchor; negations install anchors.** Giving big mode a menu of four transformation shapes made the model grab the flashiest (inversion) every time; prohibitions ("beware inversion") put the anchor in the prompt twice. Frame generative instructions positively (bigger = the destination changes) and define variety RELATIVE TO THE LIVE BOARD ("open territory the board does not yet have") so the anchor moves with the conversation instead of sitting in the prompt.
- **Models game goalposts; define the terms.** Told that every starred item must serve "the title's thread" or be left out, the model widened the title to "X AND Y" and kept everything. Commitment fields only bite when the key term is defined (a thread is ONE workflow family; a title needing "and" is two threads).
- **Verb-level bans hold where counts don't.** "No idea for a below-Adoptive team uses auto-generate/trigger/sync" and "no Big Move action starts with Build/Create/Set up" both held immediately; percentage and count instructions never did. When a rule can be expressed as banned sentence-starts or banned verbs, do that.
- **Commitment fields + server trim generalize.** After `prioritizedRules` (playbook counts), the same pattern fixed primitives counts (`focusCategories`/`stretchCategory`) and modality spread (`modalityCoverage`); required schema fields force the behavior that prose counting cannot. Make conditional requirements conditional in code (the assistant family is only required when someone can set one up); a forced field can otherwise force a floor violation.
- **Use `tool_use` for structured output, not free-text JSON.** Free-text JSON is a fragile contract: any model preamble, trailing prose, or stray brace breaks `JSON.parse`. No regex or "robust parser" fully defends that boundary. When the response must be a structured object, define a JSON Schema and force it via `tool_choice: { type: "tool", name: ... }`. Deletes the entire parsing layer and removes a whole class of intermittent failures. Workshop-load testing went from ~7/10 to 10/10 after migrating `playbook-generate.js` this way. Applies equally to `primitives-generate.js` and `synthesis-generate.js` if their text-mode parsers ever start failing.
